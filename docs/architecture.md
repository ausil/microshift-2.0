# Architecture

MicroShift 2.0 uses a separate-services model where each Kubernetes component runs as its own systemd service, rather than being embedded as Go libraries in a monolithic binary. The main `microshift` daemon acts as an orchestrator: it generates TLS certificates, writes kubeconfig files, produces environment-file-based configuration for each component, and manages their lifecycle through the systemd D-Bus API. This design leverages Fedora's existing `kubernetes1.32`, `etcd`, and `cri-o` packages directly, keeping MicroShift itself minimal and allowing components to be updated independently via the system package manager.

The systemd dependency chain ensures components start in the correct order. The `microshift.service` unit starts first, performing certificate generation and configuration writing. Once it signals readiness via sd_notify, `microshift-etcd.service` starts, followed by `microshift-apiserver.service` (which depends on etcd), and then the remaining control plane components: controller-manager, scheduler, kubelet, and kube-proxy. All component services use `PartOf=microshift.service`, so stopping or restarting MicroShift cleanly tears down the entire stack.

After the control plane is running, MicroShift bootstraps the cluster by applying embedded manifests for kindnet (CNI) and CoreDNS. It then enters a health monitoring loop, watching each component's systemd state and performing periodic health checks against the API server. Certificate rotation is handled automatically before expiry, with components restarted as needed.
